# Agent Rules

## 1. Load Order

Read these files in order before starting any task:

1. PROJECT.md       ← what this project is and why
2. ARCHITECTURE.md  ← how it is built
3. CONVENTIONS.md   ← how to write code here
4. Your task spec   ← what to do now

---

## 2. Default Mode: Discuss Before Execute

Core principle: conversation before action, always.

- Default mode is DISCUSS — never execute unless explicitly instructed
- When given any request, first propose a plan and wait for approval
- A question is not an execution request
- A design discussion is not an execution request
- Only switch to EXECUTE mode when the user says so explicitly
  (e.g. "go ahead", "implement this", "do it now")

When given any request, always do this first:
1. Restate your understanding of the goal
2. Propose a step-by-step plan
3. List any assumptions you are making
4. Wait for confirmation before touching anything

---

## 3. When to Stop and Ask

Core principle: never resolve ambiguity silently — surface it immediately.

Stop and ask when:
- Task or scope is unclear
- Instructions across docs contradict each other
- A requirement has more than one reasonable interpretation
- You are about to modify a baseline, data contract, or add a new dependency
- You discover something unexpected mid-task — do not silently fix it
- Formula or source paper is ambiguous

---

## 4. Execution Discipline

Core principle: one step at a time, always.

- Before starting, break the task into explicit steps and list them
- Complete one step, report what you did, then wait before proceeding
- After each step: "I have finished X. Ready for the next step?" 
  — always wait for confirmation
- Do not proceed to the next step unless told to
- If you notice a separate issue while working, flag it — do not fix it inline
- Never chain multiple changes in a single action
- Implement only what the formula specifies — do not optimize,
  abstract, or extend unless explicitly asked

---

## 5. Reproducibility Rules

Core principle: every experiment must be exactly reproducible
from its config alone — no exceptions.

### Configs
- Never hardcode hyperparameters, paths, or settings in code
- All experiment parameters must live in the config file
- Never modify an existing config — always create a new one
- Config filename must reflect what changed
  (e.g. lr0.001_seed42.yaml, not experiment_v2.yaml)

### Random Seeds
- Every script that involves randomness must accept a seed parameter
- Always set seeds explicitly for Python, NumPy, and PyTorch:
  random.seed(seed)
  np.random.seed(seed)
  torch.manual_seed(seed)
  torch.cuda.manual_seed_all(seed)
- Never run an experiment without a seed set

### Logging
- Every run must log: config used, git commit hash, timestamp,
  random seed, hardware info
- Never delete or overwrite existing logs
- If a run fails, the partial log must be preserved

### Dependencies
- Never import a library that is not already declared in pyproject.toml
- If a new library is needed, stop and flag it — do not add the
  import and proceed silently
- Any approved new dependency must be added using:
  uv add package==version  (e.g. uv add einops==0.7.0)
- Never edit pyproject.toml manually for dependencies
- Never manually edit uv.lock — it is auto-generated by uv

## 6. PyTorch Conventions

Core principle: never bypass the base class or Trainer — they
exist for a reason.

### Device
- Never use .cuda() or .cpu() directly
- Always pass device via map_location parameter
- All code must run on CPU, CUDA, and MPS without modification

### Model Structure
- Models own exactly two things: learnable parameters and compute_logits()
- Never put training, evaluation, or metric logic inside a model class
- Never put data processing beyond preprocess_data() inside a model class
- Always subclass BaseCognitiveModel for complete models
- Use FormulaOnlyCognitiveModel when data contract is not yet defined
- Always define config_keys — must cover every constructor argument
  needed to recreate the model

### Training and Evaluation
- Always use Trainer for training and evaluation — never call
  model.train() or model.eval() directly outside of Trainer
- Always set model and optimizer mode together as a pair:
  self.model.train() and self.optimizer.train()
  self.model.eval() and self.optimizer.eval()
- Always wrap evaluation in torch.no_grad()
- Always preprocess data through Trainer.preprocess_dataframes(),
  never call model.preprocess_data() directly

### Saving and Loading
- Always use save_parameters() to save — never bare torch.save()
- Always use from_saved() or load_saved_model() to load —
  never bare torch.load()
- Never save weights only — full parameter_payload() must always be used
